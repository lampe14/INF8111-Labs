    Not vacationing.  Just sleeping.  One is better.  I think I got the short end of that stick.
This is a nasty bug that a lot of people are complaining about.  Bugs 7770, 7773 and 8437,
for instance, (assigned to hyatt and myself) are probably all dupes of this bug, though they
sound different enough that we aren't certain enough of that to mark them as such.
  I understand everyone wants this bug fixed.  But it's a mystery why it's happening, and
it's going to be a while before any of us stumble into an answer.  We know that it's timing
dependent: if you put in pauses (like, dropping into the debugger) in strategic places, it
will work differently.  By the way, it's more a layout bug than a windowing bug, and is
almost certainly not in any code I've ever touched.  I'll eat my hat if it turns out I'm wrong
about that, and I own some big, dry, scratchy hats.
  I'm going to have to ask for patience on this bug.  People blocked by this should think
about workarounds.  It's probably not going to be fixed by M8, though I'm leaving it
targeted there for now, just in case I get lucky.

    Tested with yesterday's and today's M8 builds on Mac and Windows.

The Profile dialogs show that problem too. The first time you launch the
application, you get a blank window on Mac and a garbled display on Windows. The
workaround is to resize the dialog (which is not obvious on the Mac because the
dialog doesn't have a growbox). If you click "Continue", nothing happens on the
Mac and the display gets even more garbled on Windows. Again, a resize fixes it.

I noticed other display problems both on Mac and Windows. For instance:
- The 2 top webshells in the Apprunner sidebar don't show anything. The 3rd
webshell does work maybe because it is resized with the window.
- Test9 doesn't redraw properly when you resize the frames (especially when you
enlarge the top frame towards the bottom of the window). Resizing the window
fixes everything.
- The top frame of the phonebook (http://phonebook) does not show at all. There
is no workaround for that, certainly because the top frame consists of a single
table cell that is not resized with the window.

I'm sure we can find many similar cases. Overall, it seems to me that:
- All these problems are related.
- They don't seem to be typical of XUL dialogs as described in the summary line.
- They are serious enough to be considered blocking for M8. I'll complain loudly
if this bug is pushed to a future milestone, and try to help in the meanwhile.

Changing platform to All.
The Component should probably be changed to Layout too.
