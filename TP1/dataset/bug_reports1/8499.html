    (I'm putting this in under HTMLTables, but it may go to Layout or whomever
owns <APPLET>).

 This is a followup to bug #6990. The first problem (see bug #6247) is now
VERIFIED FIXED.

 The table(s) in this page have some 'quirky' HTML. When the above URL loads
the right hand side of the tables are clipped. In addition, if you narrow the
window, you can see this clipping continues to increase (i.e., more of the
tables contents are cut off).

 It turns out though that this problem is driven by an <applet></applet>
element in the page. If I make one change to the page -- convert to
<xxxapplet></xxxapplet> so that this is now an unknown tag pair -- then there
are *no* problems with the page layout.

 I'm going to attach two test case: the first is simply a BASE HREF'd copy of
the above URL; the second is identical to the first with
's/applet/xxxapplet/'.

 If you load the first, you can see layout reserve space for the applet
(width=440 height=27) and then, subsequently, collapse the reserved space. It
seems that the table(s) become horked on this triggered reflow.

 If you load the second, there are no layout problems (what a parser ;).

[Tested win95 jun17 viewer.exe]

    Hi Chris! I just looked at this using the most recent (Oct28) build using
WinNT.  None of the problems that I noted in June (6/99) exist (and they were
dead by at least last August).

The applet does indeed not load (from the original URL
http://www.interdynamic.gr/vacprop.asp?vacpropertyID=19)

fails to load when contained in a pathologically malformed table". In June,
the table was completely, umm, confused; _today_, it is structured correctly

since June).
