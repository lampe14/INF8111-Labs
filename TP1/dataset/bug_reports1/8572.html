GDB on Linux may have trouble obtaining
the stack trace of the pthread that crashed.
Newer releases of GDB are thread-aware, but
the Linux pthreads are actually process clones.
Each would dump its own core, so I am not sure
if GDB is able to pick up the right core file.

I would try to reproduce this crash on Solaris
and use dbx to debug.

    According to what I read in comp.programming.threads,
the gdb that comes with Red Hat 5.2 or newer is
thread-aware, but the core file format does not
support threads.  So you can run the application
from within gdb and examine any thread's stack,
but you can't debug a core file post mortem.

Here are some suggestions:
- Run the app from within gdb and try to reproduce
  the crash.
- Install a signal handler for fatal signals such
  as SIGSEGV and SIGBUS that calls pause().  Then
  when the app gets one of these fatal signals,
  it won't crash, and you attach gdb to the app
  to debug.  You might be able to use the signal
  handler in mozilla/nsprpub/pr/src/md/unix/unix.c,
  function _PR_UnixInit, if your app doesn't have
  a signal handler for SIGSEGV and SIGBUS yet.
  You will need to modify the ifdef's in that
  function to have the signal handlers installed.
- Reproduce the crash on Solaris.
