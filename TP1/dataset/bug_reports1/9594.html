    This was just published last night on Bugtraq.

Overview Description:

The "origintaing server" cookie restriction does not reject cookies set by
<SCRIPT SRC="..."> tags or CSS <LINK> tags. This is true both for direct links
<SCRIPT SRC="http://ad.doubleclick.net"> and indirect links <SCRIPT
SRC=/bin/redirectToDoubleclick">, both for HTTP documents and local files.

Steps to Reproduce:

1) enable javascript (and CSS for this CSS example). Choose the "originating
server" cookie preference.The following shows a redirected CSS hack that sets
two cookies for two separate domains.

2) create a page with code like
     <link rel="stylesheet" type="text/css" href="..."><link rel="stylesheet"
type="text/css" href="http://foo.example.org/cgi-bin/toDC">

3) on the foo.example.org server make a CGI "toDC" that looks like
  print "Set-Cookie: org=yes; DOMAIN=.example.org; PATH=/\n";
  print "Location: http://bar.example.com/cgi-bin/css\n\n";

4) on bar.example.com make a CGI that looks like
  print "Set-Cookie: com=meToo; DOMAIN=.example.com; PATH=/\n";
  print "Content-type: text/plain\n\n";
  print " /* empty CSS with comment */\n";

5) on some .example.org server and some .example.com server, make a CGI that
show the value of $HTTP_COOKIE

6) view the page you created in 2)

7) confirm the cookies by calling the CGI's in 5)

[Of course this is easier with Netscape 4.x as the warning dialogs mean you
don't *have* to do steps 5 and 7.]

Actual Results:

The browser accepts the cookies offered both by example.org (which sent no
content, only a redirect) and example.com (which is not in the same domain as
the URL of the page being viewed).

Expected Results:

The browser should have rejected both cookies. With the "originating server"
preference, the browser, in the process of assembling and rendering the page,
should only accept cookies valid for the hostname of the server that provided
the original "skeleton" document.

Build Date & Platform Bug Found:

Discovered/Verified on Netscape Communicator 4.51/4.61 on x86 NT, 4.6-0 on Red
Hat Linux x86. Verified to work with or without redirecion on <SCRIPT> or CSS
<LINK> tags, with original document fetched with http:// or file:// methods.

Additional Builds and Platforms Tested On:

Verified on Mozilla build M7 on x86 Linux.

http://securityfocus.com/templates/archive.pike?list=1&date=1999-07-8&msg=Pine.L

    What you are probably seeing is the result of redirects.  Specifically, if you 
go to site a.com and it does a redirect to b.com, then b.com is considered the 
originating server.  This is different than going to a.com and having it load an 
image from b.com.  In the former cookies from b.com are not considered foreign 
cookies whereas in the latter case they are.

If your example is not due to a redirect, then open a new bug report on it.  
Don't re-use this bug report because this one has already been morphed (started 
out talking about JS and CSS) and it's very difficult to track bugs that morph.
