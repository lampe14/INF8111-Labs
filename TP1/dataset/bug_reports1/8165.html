    The problem as I recall was that the ordinals jump around from
compile-to-compile, release-to-release, so either you need to specify them in a
.def file, or use non-ordinal-based linking (slow), or not use them at all (and
use services instead). We're not seeing this problem now because we're
building all the components at once, but if we ever want to change just
one.... Services have the added advantage that you can evolve the interface via
the different-interface--different-iid mechanism, whereas global functions you
can't do this with.

nsMsgKeySet.cpp - base utility class, needed for utility
nsMsgLineBuffer.cpp - buffering utility classes, news, local derive for utility
routines
nsMsgMailNewsUrl.cpp - implements nsIMsgMailNewsUrl, everyone derives from it
nsMsgProtocol.cpp - implements nsIStreamListener, everyone derives from it
nsMsgTxn.cpp - implements nsITransaction - utility class?
nsMsgUtils.cpp - a bunch of utility classes
nsNewsSummarySpec.cpp - subclass of nsFileSpec, can probably be moved into news

    so, in short...

http://mxr.mozilla.org/comm-central/search?string=NS_MsgGetPriorityFromString

still exists.

The idea is that every exported symbol you have (and typically NS_{foo} is exported) requires a certain kind of binding.

You have two choices for binding:
1. ordinal, this means that you have to pay close attention to your exported symbols and make sure that if you deprecate/remove something, you never let anything else fill its place, and that changing a symbol's arguments must be treated as removing it and adding something else.
2. by name, this is what we generally do, and it's relatively expensive.

It's not a big deal given where your product is in life cycle, but ideally you would fix it eventually.
